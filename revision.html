Remaining Defects to Fix

1. Live registry metadata is overwritten by definitive session registration High

Root cause: runSubprocess() applies cached workspace, toolPolicy, and routerDecision metadata to an early registry ref. Later, createAgentSession() calls AgentRegistry.register() for the same ID and replaces that ref with a fresh object.

Result: Metadata is missing while the subagent is running: isolated tasks may not appear in /workspaces, and /agents may omit router/tool policy information until terminal re-registration.

Required fix: Preserve existing metadata when registering the definitive live session, or explicitly reapply cached metadata after createAgentSession() completes. Live and terminal refs must contain consistent metadata.

2. Mission change count is not an actual changed-file count Medium

Current issue: changes is derived from the number of patch artifacts and nested patches. One patch can contain many modified files, so this does not satisfy the declared file-count meaning.

Required fix: Compute the actual changed-file count from reliable delta/patch/merge data. If reliable file-count data is not available, represent the value explicitly as unknown and update types and display formatting accordingly; do not encode unknown as an invented number.

3. Mission verification state is never connected to verification results Medium

Current issue: Missions always record verificationStatus: "none", even though verification support now exists.

Required fix: Connect mission status to real verification results when verification has been run for that mission/workspace. If no verification ran, retain "none" accurately.
